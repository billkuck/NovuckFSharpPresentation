Nice, that sounds fun. For a .NET user group, you mostly want to answer:

“Why would I, a C# dev, actually use F# in my real job?”

Here are the “top things” you could cover, organized so you can mix-and-match into a 45–90 minute talk.

1. Why F# in a C# Shop?

Goal: Set the tone: F# is for .NET devs, not against them.

Topics you could hit:

  • F# is just .NET

    • Same CLR, same BCL, same NuGet.

    • You can call C# from F# and vice versa.

  • Where F# shines compared to C#

    • Domain modeling (records + DUs) vs big object graphs.

    • Safer code (null-free by default, expression-oriented).

    • Less boilerplate for data-heavy code.

  • “Complement, not replace” story

    • F# for domains, rules engines, calculations, data pipelines, background jobs.

    • C# for UI-heavy, framework-heavy stuff if that’s where the team is comfortable.

Good opening demo:
Take a tiny C# model (e.g., Customer, LoanApplicationStatus, Nullable<decimal> for rate) and show the same in F#: record + discriminated union + option. Show how the types themselves rule out impossible states.

2. Core Language Features That Make a Difference

This is the “F# tour” section — super relevant for 45–60 minutes.

  • Records

    • Lightweight immutable types, equality out of the box.

    • Copy/update { customer with Email = … }.

  • Discriminated Unions (DUs)

    • Modeling states and workflows (New | Approved | Rejected of Reason).

    • Replace enums + flags + nulls + comments.

  • Pattern Matching

    • Exhaustiveness checks.

    • Cleaner multi-branch logic than if/else or switch.

  • Options instead of null

    • Some/None vs “maybe it’s null, maybe not”.

    • Compiler forces you to handle the missing case.

  • Pipelines (|>) and function composition

    • Turn “inside-out” code into readable top-down data flows.

    • Before/after example: nested method calls in C# vs linear pipeline in F#.

Demo idea:
Model a simple workflow, like “submit loan application → validate → score → return decision” as:

fsharp
type LoanAmount = private LoanAmount of decimal
type ApplicationStatus =
  | Draft
  | Submitted
  | Approved of approvedLimit:decimal
  | Rejected of reason:string

Then show pattern matching over ApplicationStatus.

3. Domain Modeling with Types (the “Killer Feature”)

If you only deeply cover one thing, make it this.

  • Making illegal states unrepresentable

    • Example: “Pending” approval must have a RequestedAmount, “Approved” must have a Limit, etc.

  • Encoding business rules in the type system

    • DUs for finite states, records for aggregates, smaller types for invariants.

  • Refining primitives

    • CustomerId, AccountNumber, Percentage, NonEmptyString.

    • Avoid “primitive obsession” (string/int everywhere).

  • Refactoring safety

    • Changing a DU forces you to fix all match expressions; compiler guides refactors.

Demo idea:
Take a messy C# DTO full of nullable fields and boolean flags and iteratively model it in F#. Run through “what bugs this would have caught”.

4. Practical Error Handling (without Exceptions Everywhere)

  • Using Result<'T,'Error> instead of throwing for expected failures.

  • Railway-oriented programming idea:

    • Functions that move from Input -> Result<Output,Error> and compose.

  • Pattern matching on result:

    • Logging and mapping errors in a central place.

  • Interop story:

    • You can convert F# results to C# friendly types and vice versa.

Demo idea:
Show a “validate & save” pipeline that returns Result with a list of validation errors and only touches exceptions for truly exceptional failures.

5. F# in an Existing .NET Codebase

This will really connect with the audience.

  • F# project added to a C# solution

    • Reference C# projects; expose F# functions as normal .NET classes.

  • Good starting points

    • A small service or microservice.

    • A background job or batch processor.

    • A rule engine / calculation engine (interest, pricing, risk scoring).

    • Test projects (property-based tests, domain tests).

  • Interop patterns

    • F# “core” library + C# “shell” (web/API/UI).

    • Keep F# surface area simple (records, simple DUs) for easy consumption.

You could even briefly show a solution in Visual Studio / Rider with both F# and C# projects wired up.

6. Async, Data, and Scripting

Optional section depending on how deep you want to go.

  • Async workflows and tasks

    • task {} / async {} workflows in F#.

    • Great for data pipelines and IO heavy work.

  • F# as a scripting language

    • .fsx scripts for tooling, one-off data fixes, ETL jobs.

    • Using dotnet fsi as a REPL to poke at APIs, libraries, databases.

  • Data-focused capabilities

    • (If you want) mention type providers at a high level, without going deep.

Demo idea:
Tiny .fsx script to call an HTTP API, transform the JSON, and write a CSV — with almost no ceremony.

7. Testing & Property-Based Testing

If your crowd is into tests, this could be a strong section.

  • Using F# for test projects even if production is in C#.

  • FsCheck / property-based testing concepts:

    • Instead of example tests: properties like “debits & credits must balance”.

    • F#’s concise syntax makes tests easier to read and maintain.

  • How this uncovers edge cases you wouldn’t have thought of.

Demo idea:
Take a simple pure function (e.g., interest calculation, discounting) and show 1–2 FsCheck properties.

8. SAFE Stack / Web with F# (Optional)

If you want one “wow, full stack FP” moment:

  • Quick overview: SAFE = Saturn + Azure + Fable + Elmish.

  • Same language and model shared between client and server.

  • Strongly-typed HTTP endpoints, shared DTOs, less runtime surprise.

You probably don’t want to build a full SAFE app live, but a few slides and a quick repo walkthrough could be nice if time allows and the group is web-oriented.

9. Tooling & How to Get Started

Wrap up with a “getting started” slide:

  • Tooling:

    • Visual Studio, Rider, VS Code + Ionide.

  • Project templates:

    • dotnet new console -lang F#, test templates, SAFE templates.

  • Learning resources:

    • Books, community sites, sample GitHub repos.

  • Suggested first uses:

    • “Pick one small service, report, or job and try rewriting it in F#.”
